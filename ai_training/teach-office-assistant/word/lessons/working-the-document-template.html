Word to a High Standard · Lesson 0009

Working the document template

Tied to your mission: your mission states it plainly — "everyone starts from the right document." New documents are born from the template, not from last month's copy. Much of what you produce goes out under the department's name, so every document you send is a small act of information disclosure whether you mean it to be or not, and the fastest way to disclose the wrong thing is to start a new one from an old one. This lesson is about the first thirty seconds of a document's life, because everything wrong with it later traces back to how it was born.

Practice pack · open this first

Open the practice pack and find Scenario E — The new-doc job: a supplier-onboarding pack is needed, the org template exists, and three live documents already carry a 2024 footer and a previous supplier's name in their metadata because someone took the faster-looking shortcut.

The one idea

A new document should be born from the template, never from last month's file. New-from-template gives you exactly the template's styles, its doc-control footer, and clean document properties — nothing else. Save-As on a predecessor gives you all of that too, but riding along invisibly: the 2024 footer, the previous supplier's name sitting in the file properties, a superseded clause nobody remembered to delete. The predecessor is not a shortcut to the same starting point — it is the template plus every mistake anyone has ever made in it, inherited silently.

The test that tells you which one you actually got is the same every time: Inspect Document. Run it before anything ships. A document born from the template comes back clean. A document born from Save-As comes back carrying a ghost — a name, a date, a comment — that was never meant to travel to this recipient at all.

Why it matters to an office assistant

Onboarding packs, meeting minutes, training material, department procedures — you produce the same kind of document again and again, often within days of each other, and most of it goes out under the department's name rather than yours alone. That repetition is exactly the setup where Save-As inheritance turns into a disclosure incident: build this month's onboarding pack from last month's file, change the visible names and dates, and the file properties still say the old supplier's name until someone checks. A new supplier opening that pack and finding a previous supplier's name in Author or in a stray comment isn't a formatting slip — it's the kind of thing that comes back to the department, not quietly to you.

The people who never have this problem aren't more careful proofreaders. They simply never start a new document from an old one — every document they send began its life from the template, so there is nothing previous to leak.

The bad habit this undoes "Grab the last one and change the names." It's faster for the first ten minutes — the layout's already there, the boilerplate's already written — and then Riverside's name surfaces in the file properties of GreenField's onboarding pack, read by GreenField. The habit feels efficient because the visible text looks right immediately; it fails silently in the metadata, the comments, and the hidden text nobody scrolls down to see.

How to do it

Work Scenario E. A supplier-onboarding pack is needed for a new supplier. Do not open any previous onboarding pack. Start from File → New and choose the organisation's template (or open the template file itself and immediately Save As under the new pack's name — never open a predecessor document and Save As from there; the difference is entirely about whose history you inherit). The template-born file already has the cover page, the styled heading levels, and the doc-control footer built in — you did nothing to earn those, they came free with the correct starting point.

Use the template's styles as given. If the pack needs a heading level or a callout style the template doesn't define, that is not licence to invent one locally — a hand-built substitute today is exactly the kind of local patch that becomes next year's inherited mistake. Treat a missing style as a request to the template's owner, the same discipline the deck-building side of this course applies to layouts (see Working the corporate template for the same contract on slides). Write down what's missing and who owns the template; don't paper over the gap yourself.

Now run the ghost-hunt drill, because the habit only teaches you something if you see the wreck it leaves. Take a Save-As copy of a previous onboarding pack — the fast, wrong way to start — and open File → Info → Check for Issues → Inspect Document. Run it against Document Properties, Comments, and Hidden Text at minimum. Read what comes back: the predecessor's author name, a reviewer's comment thread that was never meant for this recipient, maybe a hidden paragraph from a clause that got "deleted" by hiding rather than removing. That is what would have shipped to the new supplier if nobody had checked.

Finally, run the identical Inspect Document pass on the template-born file you started this exercise with. It comes back clean — no author but you, no stray comments, no hidden text. That contrast, on the same day, on two files built from the same content, is the proof: the starting point decided the outcome before you typed a single word.

Try it for real

Start a real document you actually need this week using File → New from your organisation's template — not a copy of any previous file, even one of your own that you're proud of. Build it using only the template's own styles, flagging anything missing rather than inventing it locally. Self-check: run Inspect Document (File → Info → Check for Issues → Inspect Document) against Document Properties, Comments, and Hidden Text before you consider it done. It should come back clean. If anything from a previous document surfaces — a name, a date, a comment — you started from the wrong place; go back to the template and rebuild.

The AI moves — climbing the ladder

Same task — start a new document from the template and populate it without fighting it — with more handed to AI at each rung. The check never changes: does every part of the document trace to a real template part (or an explicit request to the template's owner), and does Inspect Document come back clean.

Rung 2 · Aldi:GPT — it maps the document to the template, you judge the map

Aldi:GPT can't see your template file, so it can only work from what you describe; you judge whether the section it names for each part of the document actually exists in your template and actually fits the job.

Type this prompt into Aldi:GPT

Check: the signature failure at this rung is template-fighting — proposing local styles or structures the template already provides, like inventing a new heading style when Heading 2 already does that job. Every mapping in its answer must name a real template part or flag an explicit template-owner request; an answer that quietly invents a workaround has failed the check even if the resulting document would look fine.

Rung 3 · M365 Copilot in Word — it drafts inside the template, it doesn't replace it

Copilot in Word can draft the new document's section content from a prompt, but it only stays on-template if you're drafting inside the template-born file already — it writes text into whatever structure is open, it does not create a template of its own accord.

Type this prompt into Copilot (in Word, with the template-born file open)

Check: click into Copilot's output and look at the Styles gallery — did the new section inherit the template's Heading 2 and Body styles, or did it arrive with its own direct formatting riding on top? Same check as the styles lesson; it compounds here, because a section that arrives direct-formatted is a second, quieter way for template-fighting to sneak back in, even when you never touched a predecessor file.

Rung 4 · GitHub Copilot — it builds the template itself, from a brand/doc-control spec

An agent with file access can generate the organisation's template file itself via python-docx — styles, footer fields, and document properties — from a written spec, but it can't infer your brand or doc-control rules from nothing; you hand it the spec explicitly.

Type this prompt into GitHub Copilot

Check: every document born from the generated template must pass Inspect Document clean — no author, no stray properties, no residual text — straight out of the script. Then change one thing in the spec (say, the heading colour) and regenerate: the change must flow through to a freshly-born document without hand-editing. The template is code now; a style that doesn't regenerate through, or a document that fails Inspect Document out of the gate, is a bug in the generator, not a one-off to patch in Word.

The point of the ladder From "AI maps sections onto the template's parts" to "AI builds the template itself from a spec" — the deliverable never changes: a document whose every part traces to the template, and whose properties come back clean under Inspect Document. What protects you at every rung is that same check, not the tool. Learn to start from the template by hand first, so you can tell a real template-part mapping from a workaround, and a clean document from one still carrying a ghost — whether the ghost came from a predecessor file or from an AI that quietly reached for its own structure instead of yours.
