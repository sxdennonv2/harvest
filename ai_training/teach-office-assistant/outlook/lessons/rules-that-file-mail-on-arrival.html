Outlook to a High Standard · Lesson 0002

Rules that file mail on arrival

Tied to your mission: a mailbox with one home per mail, "maintained by rules rather than willpower" is the success line this course is graded against — and willpower is exactly what runs out by Wednesday. Folders, categories & flags gave every mail a fate by hand; this lesson makes the fates that never change file themselves, so the only mail left for your hands is the mail that actually needs them.

Practice pack · open this first

Open the practice pack and return to Scenario A — The Monday inbox — the same twelve messages, now examined for which ones deserve a standing rule instead of a one-off decision.

The one idea

A folder without a rule is a chore. You built the folders in Folders, categories & flags — now ask, for each one, whether the filing decision itself ever changes. A rule does the filing for mail whose right fate is identical every time: newsletters, system notices, standing reports. It is never for mail that needs your eyes first.

The test is blunt on purpose: if you'd make the same triage decision 100 times out of 100, automate it. If the decision depends on what the mail actually says — is this an FYI or a question, a notice or a chase — it stays human. A rule that can't pass that test isn't saving you a decision, it's making one for you in the dark.

Why it matters to an office assistant

The same handful of standing reports, the same IT comms, the same weekly newsletter — a department inbox repeats itself more than most, because so much of what lands there is routine correspondence that follows the department everywhere. That repetition is exactly what rules are for: every Monday you don't have to remember to drag the outage notice out of the inbox, because it never reached the inbox in the first place. What's left when the routine mail files itself is the inbox as it should be — a chase that needs chasing, a question from the team, the department's own FYI — the handful of mails each day that actually need a decision from you, not buried under forty that never did. That's the difference between running the department's mail flow and being run by it.

The bad habit this undoes Two habits, opposite poles, same waste. One: hand-filing the identical newsletter into the identical folder every single morning, as if it might be different this time. Two, the more dangerous one: over-ruling — building a rule so broad it sweeps mail that sometimes carries a real ask straight into a folder unseen, where the ask dies quietly because nobody with eyes ever saw it arrive. A rule that misfires on silence is worse than no rule at all.

How to do it

Go back through Scenario A's twelve mails and sort them a different way this time — not by fate, but by whether the fate is always the same. Mail #3 (IT Comms' planned outage notice) passes: it is dated information, never a question, every single instance of it gets filed straight to an IT-notices folder. Mail #6 (the newsletter) passes for the same reason — it's read-once-or-never, and no newsletter has ever asked you a question back. Both are candidates for a rule.

Now the never-rule pile. Mail #7 (Alex Moreau's capacity question) fails the test immediately — most of Alex's mail might be routine, but this one needs an answer by Tuesday, and a rule can't read that a subject line hides a question this time. Mail #10 (Sam Lee's chase) fails for the same reason: Sam Lee also sends the Q3 price list (#1), which is rule-safe, but a rule keyed only on sender would catch the chase too and file it unseen — the one mail in the whole set you owe by Friday. Sender alone is never enough; the rule has to be narrow enough to catch only the mail that's actually always-the-same.

Build two real rules from this. Rule 1 — condition: from IT Comms; action: move to the IT-notices folder, mark as read. Rule 2 — condition: from the newsletter's sender address, subject contains the newsletter's fixed title; action: move to a Reading folder. Notice what Rule 2 needed that Rule 1 didn't: a subject condition as well as a sender condition, because "from the newsletter" alone would also catch a support reply if that sender ever sent one.

Last, the part people skip: rules run in an order, top to bottom, and each one can carry a "stop processing more rules" action. If Rule 1 fires and stops processing, a later rule that might have moved the same mail elsewhere never runs. Put your narrowest, safest rules first, and only add "stop processing" where you've checked that no rule beneath it should ever also apply to that mail.

Check yourself

Answer these before moving on — immediate feedback below each.

1. Sam Lee sends both the always-safe Q3 price list and the sometimes-urgent promo chase. What does this tell you about a rule keyed on "from: Sam Lee" alone?

2. Which of Scenario A's twelve mails is the cleanest candidate for a standing rule?

3. You have two rules that could both match the same mail. What governs which one actually files it?

Try it for real

Find one thing in your own inbox you've hand-filed at least three times this month — the same sender, the same folder, every time, no exceptions. Before you build the rule, say out loud why it passes the 100-of-100 test: name the condition narrow enough that it could never also catch a mail from that sender that actually needed you to read it first. Build the rule, then check yourself the way you checked Sam Lee above — list every other kind of mail that sender has ever sent you, and confirm none of them would be swept up by the same condition. If you can't rule out every exception, narrow the condition until you can.

The AI moves — climbing the ladder

Same task — decide which recurring mail earns a standing rule, and build it — with more handed to AI at each rung. The check never changes: every rule that survives must pass the 100-of-100 test by name, for a condition narrow enough to prove it.

Rung 2 · Aldi:GPT — it designs the rule set, you audit every rule for over-ruling

Aldi:GPT can't see your mailbox, so it works only from the recurring mail types you describe; it designs conditions and actions, but it has no way to know which senders sometimes break their own pattern — that check is yours.

Type this prompt into Aldi:GPT

Check: the signature failure at this rung is over-ruling — a rule proposed for mail that sometimes needs action, dressed up as routine because the sender is repetitive. Read the returned rule set line by line: every rule must carry its own why-safe sentence, and the Finance accruals request must come back flagged as a never-rule, not a rule. If a rule is missing its 100-of-100 justification, reject it before it goes anywhere near Outlook.

Rung 3 · Microsoft 365 Copilot — it identifies rule candidates, it cannot build the rule

Copilot in Outlook cannot create a rule for you — be honest about that boundary up front. Its honest move at this rung is narrower and more useful: reading your actual mail history to surface which senders were pure information you never once replied to.

Type this prompt into Copilot (in Outlook)

Check: Copilot is identifying candidates by content pattern — you never replied — not by your obligations, and those are not the same thing. A sender you never replied to might still be one where the next mail needs a reply; "never so far" is not "never possibly". Audit its list against the eyes-test yourself, mail by mail, before you build a single rule from it — Copilot's list is a shortlist, not a verdict.

Rung 4 · GitHub Copilot — it builds the rules kit, it cannot touch your mailbox

An agent with file access can build you a ready-to-apply rules kit from a description of your mail types, but it cannot configure your live mailbox — every rule still gets created or scripted by you, in Outlook or via the steps it hands you.

Type this prompt into GitHub Copilot

Check: read the kit against your own description before applying anything. It must not invent a rule for a sender you never named, and every rule in the kit must ship with its why-safe justification, not just a condition and an action. A kit that manufactures rules for senders you didn't describe has failed, no matter how well-formed the PowerShell looks — send it back.

The point of the ladder From "AI designs the rule set" to "AI builds the whole kit" — the deliverable never changes: rules for the mail whose fate never varies, and nothing else. What protects you at every rung is the same 100-of-100 test, named out loud for every rule, not the tool that proposed it. Learn to spot over-ruling by hand first, so you can catch it in an AI's rule set as fast as you'd catch it in your own.
