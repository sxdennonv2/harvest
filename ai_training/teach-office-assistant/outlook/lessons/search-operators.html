Outlook to a High Standard · Lesson 0004

Search operators: describe the mail, don't scroll

Tied to your mission: you're the person who can "file, flag and find anything on request" for the department — and search operators are the manual-craft version of that promise, the one you can use in ten seconds without opening anything AI. Before you climb the ladder, learn to describe the mail to the index instead of scrolling past it with your eyes.

Practice pack · open this first

Open the practice pack and read Scenario B — The lost attachment. It's practised in your own mailbox — the pack describes three retrieval targets; you build the operator query for each before you touch the search box.

The one idea

Scrolling is searching with your eyes. Outlook already has an index built for describing the mail, not eyeballing it — operators like from:, hasattachments:, received:, flagged and unread let you tell the index who sent it, what it carried, roughly when, and what state it's in — then combine two or three to narrow a mailbox of thousands down to a handful.

The habit that makes this work: build the query before you look, from what you actually remember. You rarely remember a subject line. You remember the sender, a rough month, and "it had the attachment." Write that down as from:sam hasattachments:yes received:>=1/3/2026 received:<=30/4/2026 before you open the search box — not a half-remembered phrase you hope is in the subject.

Why it matters to an office assistant

Someone stops by your desk and asks "which price list did Harvest Lane send before the change, and when?" — the kind of request that lands on your desk precisely because you're the one who's supposed to be able to find anything. The attachment is buried under hundreds of newer messages, the subject line was never memorable, and scrolling back three months of mail to find it is exactly the kind of afternoon nobody has time for. An office assistant who can write from: + hasattachments:yes + a date range answers that question in under thirty seconds — with a real message and a real timestamp to show, not a guess. That's the difference between being the person who "will have a look" and the person who already has the answer.

The bad habit this undoes Scrolling and squinting through months of mail looking for something you'll "recognise when you see it" — or typing a misremembered subject phrase into the search box, getting nothing back, and concluding that search "doesn't work." Search didn't fail; the query never described anything the index could match on. A subject guess is one weak signal. Sender, attachment state and a date range are three strong ones, combined.

How to do it

Take the three retrieval jobs in Scenario B of the practice pack, and for each one, write the query in your head before opening Outlook's search box:

Job 1 — the price-list attachment. You remember the sender and roughly when, not the subject. Query: from:sam hasattachments:yes received:>=1/3/2026 received:<=30/4/2026. from: narrows to one contact, hasattachments:yes drops every message without a file, and the received: range boxes in the roughly-three-months-ago memory. Three operators, three narrowing cuts.

Job 2 — the unanswered question. You remember one distinctive word from it and roughly who sent it. Query: that word plus from: the colleague — the free-text word does the narrowing this time, from: keeps it inside one person's mail.

Job 3 — everything still flagged from before last month. Query: flagged received:<1/6/2026 — the state operator finds every follow-up you still owe, the date range drops the ones you already cleared.

Run each query in your own mailbox's search box (type it directly — Outlook accepts the operator syntax as typed text). Per the pack's check: each result set should fit on one screen (under 20 messages), and you should be able to name out loud which operator did the narrowing.

Check yourself

Answer these before moving on — immediate feedback below each.

1. You remember a supplier sent you a price list about three months ago, but not the subject line. What should you build the query from first?

2. A query returns zero results. What does that tell you?

3. Why combine from: with hasattachments:yes and a received: range, rather than using just one?

Try it for real

Pick your own real "lost" item from this month — a message you'd struggle to find right now. Before opening the search box, write the query on paper or in a note: name the sender, the attachment state if relevant, and a rough date range or flag/read state. Then type it into your mailbox's search box. Check: found in under 30 seconds, without scrolling — and you can point at which operator did the narrowing. If it takes longer, the query was missing a strong signal (usually the sender or the date range) — add it and re-run rather than falling back to scrolling.

The AI moves — climbing the ladder

Same task — retrieve one specific mail without scrolling — with more handed to AI at each rung. The check never changes: does the query (or the tool acting on your behalf) land on a small, nameable result, verified against the actual message rather than trusted on faith?

Rung 2 · Aldi:GPT — it writes the query, you verify the operators

Aldi:GPT can't see your mailbox, so it can only write the operator query from what you tell it — you place it in the search box and verify every operator is real Outlook syntax before you trust it.

Check: the signature failure at this rung is an invented or wrong-syntax operator — something plausible-looking that Outlook doesn't actually support, or the wrong date format. Verify every operator it hands you against the Microsoft search-operators reference before you trust the query — don't paste blind.

Rung 3 · M365 Copilot — it searches and summarises across your mailbox

Copilot in Outlook can act on the mailbox you have open and answer in natural language — "find the thread where the supplier confirmed the price change" — rather than you writing the operator query yourself.

Type this prompt into Copilot (in Outlook)

Check: Copilot will hand you back an answer or a summary — insist on opening the actual source message before you act on it. The failure here is trusting the paraphrase when the real mail says something subtly different (a different volume, a different date, a condition Copilot's summary dropped). The summary is a pointer, not the record.

Rung 4 · GitHub Copilot — it builds a mailbox report from an export

GitHub Copilot is an agent with file access but no connection to your live mailbox — given an exported CSV of mail metadata, it can build a report (senders by volume, attachment counts, ageing flags) that would take you an age to tally by hand.

Type this prompt into GitHub Copilot

Check: the totals must reconcile — the per-sender message counts must sum to the export's total row count, exactly (the same parts-sum-to-whole discipline as any report). A report that double-counts a thread across its messages, or drops rows silently, fails the check even if the numbers look plausible.

The point of the ladder From "you write the query" to "an agent reports on an export" — the deliverable at every rung is the same: a small, verifiable answer instead of a scrolled guess. What protects you is the check — real operators verified against the source, the actual message opened instead of the paraphrase trusted, totals reconciled instead of taken on faith. Craft first, so you can supervise.
