PowerPoint to a High Standard · Lesson 0002

Placeholders, not text boxes

Tied to your mission: a deck built on the master only fixes-once-fixes-everywhere if the text on every slide is actually plugged into that master's skeleton. Builds on Masters & layouts — that lesson got the deck onto one master with the right layouts; this one makes sure the text on those slides is wired into them, not just sitting on top.

Practice pack · open this first

Open the practice pack and find Scenario C — The franken-deck — the consolidated range-review deck, now on one master, but still carrying text that never made it into a placeholder.

The one idea

A placeholder is a slot the layout built for you: it inherits position, font, size, and colour from the layout and the master, and it keeps inheriting — change the master, and every placeholder on every slide built from it updates without you touching a slide. A text box is an orphan. Nobody built it a home; it inherits nothing; whatever font, size, and position you gave it by hand is the font, size, and position it will have forever, until you hand-edit it again.

A deck full of text boxes has quietly opted out of its own skeleton. Outline view goes blank where a text box should be. A layout swap leaves the text box exactly where it was, stranded on the new layout. Reset does nothing to it, because Reset only knows how to put placeholders back — it has no idea a text box exists. The test is one click: select the text and look at the border. Does it say placeholder — this came with the layout — or does it say text box — someone drew this by hand?

Why it matters to a specialist analyst

The range-review deck gets rebuilt every month from whatever's easiest to grab — last quarter's slides, a colleague's deck, a chart pasted from an email. Every time a sustainability metric or a category number lands as a hand-drawn box instead of a placeholder, that slide becomes a small island: the title fix that landed on the master last week won't reach it, the footer date correction won't reach it, and outline view — the fast way to check a whole deck's message before it goes in front of leadership — will show a gap where that slide's content should be. One franken-deck full of orphaned text boxes means every "fix it on the master" promise from the last lesson quietly breaks slide by slide, and the figures you pulled and checked once have to be re-checked by hand every refresh.

The bad habit this undoes Reaching for Insert → Text Box as the default way to get any words onto a slide — a stat, a caption, a title that didn't quite fit — instead of checking whether the layout already has a placeholder for it. Each text box drawn this way is a small, permanent, hand-styled orphan: invisible to outline view, unreachable by the master, and yours to restyle by hand for as long as the deck lives.

How to do it

On Scenario C's consolidated deck, go slide by slide and click every block of text. The selection border tells you what you're looking at: a placeholder border reads "Click to add text" when empty and is labelled by its placeholder type in the Selection Pane (Title, Text Placeholder, Content Placeholder); a text box border is just a plain box with no placeholder identity at all. Wherever you find a text box carrying real slide content — a stat, a heading typed over the template's title, a footer note retyped by hand — cut its text out.

Then check the slide's layout (Home → Layout) for a placeholder that fits what you just cut. If one exists, click into it and paste the text in. If the current layout genuinely has no placeholder for that content — say the slide needs a second text block the layout doesn't offer — switch the slide to a layout from the master that does provide it, rather than drawing a box to compensate. Once the text is sitting in a real placeholder, select the slide and hit Reset: position, font, and size should snap to whatever the layout defines, because now there's an inheritance path for Reset to use. Delete the emptied text box last, so you're not working from a slide with duplicate content mid-edit.

Finish by switching to outline view (View → Outline). Every slide's title and body text — the content you just moved into placeholders — should now appear in the outline. Any slide still showing blank in outline view still has an orphan on it somewhere; go back and find the border that says text box.

Check yourself

Answer these before moving on — immediate feedback below each.

1. You click a block of slide text and its border reads "Click to add text" with a Selection Pane label of "Text Placeholder". What is it?

2. A slide's title looks fine, but outline view shows a blank line where that title should be. What does that tell you?

3. You hit Reset on a slide hoping to snap a stranded stat box back onto the layout's position, but nothing happens. Why?

Try it for real

Work every slide of Scenario C's consolidated deck: click each piece of text, read its border, and move anything living in a text box into a real placeholder (switching layouts first if the current one doesn't offer the right slot). Hit Reset on each slide once its content is in placeholders. Self-check when you're done: switch to outline view and confirm every slide's title and body text shows up there in full — a slide that still shows blank or partial in outline view still has an orphaned text box on it somewhere.

The AI moves — climbing the ladder

Same task — get every slide's text off hand-drawn boxes and into real placeholders — with more handed to AI at each rung. The check never changes: outline view must show the deck's full text, and every border you click must read placeholder, not text box.

Rung 2 · Aldi:GPT — it reasons from what you paste, not from the deck

Aldi:GPT can't see your slides or click a border, so it can only diagnose from the outline-view text you paste it; blank lines in that paste are the tell, but confirming which shape is actually behind each blank is still a job only you can do with a mouse.

Rung 3 · M365 Copilot — new content only lands in a placeholder if the layout has one

Copilot in PowerPoint writes content straight onto the slide's layout, so if the layout already offers the right placeholder, the generated text lands in it and inherits properly; if the layout doesn't offer one, Copilot still has to put the text somewhere, and it will draw a text box to do it.

Type this prompt into Copilot (in PowerPoint)

Check: click every text block Copilot just generated and read its border. If it reads placeholder, inheritance is live and you're done. If it reads text box, the layout you're on genuinely had no slot for this content — the fix is to add or switch to a layout that provides one, not to leave Copilot's box as-is and restyle it slide by slide.

Rung 4 · GitHub Copilot — it audits placeholder vs. free text across the whole deck

An agent with file access can read the deck's XML directly and tell you, shape by shape, which text sits in a real placeholder and which is a free-floating text box — far faster than clicking every border yourself — but it cannot fix your deck; you still do the moving and the layout switching.

Type this prompt into GitHub Copilot

Check: run it against the franken-deck first — it should flag orphaned text boxes on the slides you already know are broken, slide by slide. Then run the same script against a clean, all-placeholder skeleton deck — it should come back silent, flagging nothing. A report that flags on both, or ever confuses a placeholder with a text box, is just counting shapes; it isn't testing inheritance, and you shouldn't trust its verdict on the real deck.

The point of the ladder From "diagnose the gaps I paste you" to "audit the whole deck's file structure" — the deliverable never changes: every slide's text living in a placeholder, outline view showing all of it, Reset actually doing something. What protects you at every rung is the same border-click and the same outline-view check, not the tool. Learn to tell a placeholder from a text box by hand first, so no rung above it can quietly hand you back a deck of orphans.
