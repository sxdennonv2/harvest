Word to a High Standard · Lesson 0005

Tables that behave: widths, headers, repeat rows

Tied to your mission: a range-review submission lives or dies on its supplier-comparison table, and tables that hold their shape across edits is a named line in this course's mission. A table you have to fight every time someone edits a cell isn't a table doing its job — it's a decision you forgot to make, resurfacing as a chore.

Practice pack · open this first

Open the practice pack and find Scenario C — The drifting table: the submission's supplier-comparison table, five suppliers by six criteria, misbehaving in three specific ways.

The one idea

A table behaves when you tell it to. Left to its defaults, a Word table negotiates its column widths with whatever you type into it — that negotiation is what you're calling "drift" when a long supplier name shoves every other column sideways. Three orders end the negotiation for good: fixed column widths (AutoFit switched off, so the widths are a decision you made, not an outcome the table arrived at), the header row set to repeat across pages (a property you turn on once, not a copy of the header row you re-paste every time it "falls off"), and real tables for anything tabular — tab-spaced text that merely looks like columns is a fake table, and it shatters the first time anyone edits a line above it.

This is the same demand the Excel course's Tables lesson made of a range: data as data, structured as what it is, not dressed up to look structured while secretly being loose text or an accident of formatting.

Why it matters to a buyer

The supplier-comparison table is the page a director actually reads before signing off a range review — five suppliers, six criteria, and whichever row has the lowest landed cost. If the columns drift every time someone fixes a typo in "BrightHarvest Produce Ltd", the table looks unfinished even when the numbers are right, and a reviewer who doesn't trust the formatting starts doubting the figures too. A header that vanishes on page 7 turns a comparison into a guessing game — which column was "lead time" again? A submission that survives a proof-pass with its table intact reads as a document someone controlled; one that needs re-fixing after every review round reads as one that's controlling you.

The bad habit this undoes Wrestling the drift instead of ending it: dragging column borders back into place after every edit, because AutoFit is still switched on and renegotiates the widths the moment anyone types a longer name. Re-pasting the header row onto page 7 where it "fell off" — which doesn't fix anything, it just creates a second header that is really just an ordinary row wearing bold, invisible to the repeat-header setting and destined to drift out of sync with the real one at the next edit.

How to do it

Open Scenario C's supplier-comparison table and discipline it against its own spec, one order at a time. First, the widths: select the table, open Table Layout → AutoFit → Disable AutoFit, then set each column's width deliberately by dragging or by exact measurement in Table Properties → Column — enough for the widest realistic supplier name, not the widest name currently in the table. This is the difference between a width and an accident: a width survives the next edit because you decided it, not because nothing longer has been typed yet.

Second, the header: click into the top row, open Table Layout → Repeat Header Rows (or Table Properties → Row → "Repeat as header row at the top of each page"), and confirm it's ticked. Do not touch page 7 at all — if the setting is on, there is nothing there to fix, because Word draws the header for you on every page the table crosses. If a duplicate header already exists on page 7 from a previous "fix", delete that row now; it's not a header, it's a row that will drift out of step with the real one.

Third, the fake columns: below the main table, the summary line that looks like two columns is tab-spaced text. Select it, Insert → Table → Convert Text to Table, telling Word to split on tabs. Confirm it lands as a real table with its own cells, not text that merely lines up today.

Now the test, typed live: click into the shortest supplier name and retype it as something absurdly long — "BrightHarvest Sustainable Produce Group (Northern Region) Ltd" — and watch. Nothing else moves. The cell wraps to a second line inside its fixed width; no column border shifts; the header is exactly where it was.

Check yourself

Answer these before moving on — immediate feedback below each.

1. Every time a supplier name gets edited, the table's columns resize themselves and you drag the borders back. What's actually happening?

2. The header row keeps "vanishing" on page 7, so you paste a copy of it there each time. What's wrong with that fix?

3. A summary line under the table lines up into neat columns using the Tab key. Why is this a problem waiting to happen?

Try it for real

On a table in your own work (or Scenario C's, if you don't have one to hand), apply all three orders: AutoFit off with widths set by decision, header row set to repeat, any tab-spaced pseudo-table converted to a real one. Self-check: type a supplier or product name at least twice as long as anything currently in the table, into any cell. If any column border moves, or the header duplicates or disappears on a later page, one of the three orders isn't actually applied yet — go back and check Table Layout again before you call it done.

Rung 3 · Microsoft 365 Copilot — honest boundary: the properties are native, Copilot drafts around them

Table Layout, AutoFit, and repeat-header are native Word controls — no AI is involved in setting them, and none is needed. Copilot's real move in Word is drafting and summarising text around a table, or formatting consistently on request; be honest that fixing the table's own properties is still done by hand, in the Table Layout tab.

Check: after any Copilot move that touches a document containing a table, verify every number and every supplier name in the table survived verbatim — models resist rewriting a table less than they should, and a "helpful" pass can quietly reorder rows or round a figure. This is the proof-pass discipline: read the table cell-by-cell against the source before you trust it.

Rung 4 · GitHub Copilot — it builds the table from the data files

An agent with file access can generate the table directly from your source data with python-docx — fixed widths from a spec, repeat-header set programmatically, one real table per data source, rather than typed and fought with by hand.

Check: reconcile every total and figure in the generated table against the source data file — parts to whole, nothing dropped or rounded in transit. Then regenerate with one supplier name swapped for a much longer one: confirm the diff touches only that cell's text, and no column width, row, or other cell changes as a result.

The point of the ladder From "AI names which property to change" to "AI builds the table programmatically from data" — the deliverable never changes: a table with fixed widths, a repeating header, and no fake columns pretending to be real ones. The check that protects you at every rung is the same one from rung 1: type a long name in, and see whether anything moves that shouldn't. Learn to discipline a table by hand first, so a generated one can be verified against a standard you already know how to apply yourself.
